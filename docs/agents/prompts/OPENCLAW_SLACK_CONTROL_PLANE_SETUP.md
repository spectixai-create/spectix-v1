# OpenClaw Slack Control Plane Setup

## Purpose

Use one private Slack channel as the first real OpenClaw control plane for
Spectix agents:

- `ceo`
- `architect`
- `pm`
- `qa`
- `codex_ops`

This is setup preparation only. It does not enable real routing, start 24/7,
enable cron, expose public webhooks, merge, deploy, mutate data, run smoke, or
store credentials in git.

## Slack Workspace Requirements

- Use a private Slack channel dedicated to Spectix OpenClaw control.
- Restrict channel membership to approved operators only.
- Invite the OpenClaw bot only to the private control channel.
- Do not invite the bot to public channels for first activation.
- Do not post production credentials, Supabase keys, Slack tokens, Vercel
  tokens, or claim PII in Slack messages.
- Use Slack threads for individual task discussions once basic dummy routing
  passes.

## Slack App Setup Steps

1. In Slack API, create a new app.
2. Prefer "From manifest" if the OpenClaw setup wizard provides a manifest.
   Otherwise create from scratch.
3. Enable Socket Mode.
4. Create the app-level token with the Slack `connections:write` capability.
   Store it locally as `SLACK_APP_TOKEN`.
5. Configure OAuth bot scopes.
6. Enable Event Subscriptions for Socket Mode events.
7. Configure slash command `/openclaw` if using OpenClaw native slash commands.
8. Enable App Home tabs/messages if DM control is needed later.
9. Install the app to the workspace.
10. Copy the bot token locally as `SLACK_BOT_TOKEN`.
11. Create a private channel, for example `#spectix-openclaw-control`.
12. Invite the bot to that private channel only.
13. Record the Slack control channel ID as `SLACK_CONTROL_CHANNEL_ID`.
14. Record approved operator Slack user IDs as `ALLOWED_SLACK_USER_IDS`.

## Credentials Needed

Names only:

- `SLACK_BOT_TOKEN`
- `SLACK_APP_TOKEN`
- `SLACK_SIGNING_SECRET` if HTTP Events API is ever enabled later
- `SLACK_CONTROL_CHANNEL_ID`
- `ALLOWED_SLACK_USER_IDS`

## Where Credentials Must Live

- Local OpenClaw config or local secret store only.
- Environment variables are acceptable if the shell/profile is local and not
  committed.
- Never git.
- Never docs.
- Never `.env.local` unless OpenClaw explicitly requires it and the file remains
  ignored.
- Never Slack messages.

## Allowed Initial Commands

Only allow read-only or local dispatcher inspection commands at first:

```text
/oc status
/oc handoff TASK-SPECTIX-001 --role pm
/oc handoff TASK-SPECTIX-001 --role qa
/oc next
/oc audit
```

These commands should read OpenClaw/local dispatcher state and produce
non-mutating responses. They must not create claims, upload documents, mutate
Supabase, merge, deploy, or edit app code.

## Forbidden Commands

Block or refuse:

- merge
- deploy
- production
- DB migration
- secrets
- env changes
- create claim
- upload docs
- smoke execution
- cron
- 24/7
- auto-run
- auto-merge
- auto-deploy

## Dummy Validation Procedure

Do not run until CEO approves local Slack credential entry.

1. Confirm no repository files are dirty:

   ```powershell
   git status --short
   ```

2. Validate local OpenClaw config:

   ```powershell
   openclaw config validate
   ```

3. Confirm cron remains disabled:

   ```powershell
   openclaw config get cron
   ```

4. Confirm Slack channel config exists locally without printing token values.

5. Start the gateway in foreground loopback mode only:

   ```powershell
   openclaw gateway run --bind loopback --auth none --ws-log compact
   ```

6. In the private Slack channel, send:

   ```text
   /oc status
   ```

7. Send:

   ```text
   /oc next
   ```

8. Send:

   ```text
   /oc handoff TASK-SPECTIX-001 --role pm
   ```

9. Verify:
   - command replies appear only in the private control channel/thread
   - output is read-only
   - no repo files changed
   - no app code changed
   - no database rows changed
   - no secrets printed
   - no cron or 24/7 service started
   - no merge/deploy attempted

10. Stop the foreground gateway.

11. Re-check:

    ```powershell
    git status --short
    openclaw config get cron
    openclaw agents bindings --json
    ```

## Rollback / Disable

If Slack routing behaves unexpectedly:

1. Stop the foreground gateway process.
2. Remove or disable Slack credentials in local OpenClaw config.
3. Remove the bot from the private control channel if needed.
4. Confirm `cron.enabled = false`.
5. Confirm no OpenClaw service is installed or running.
6. No repository cleanup is required for local credential removal.

## Activation Boundary

This setup package ends before real routing is enabled. The next CEO-approved
task must explicitly authorize local credential entry and dummy Slack validation.
