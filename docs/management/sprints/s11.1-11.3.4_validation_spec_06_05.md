## 1. Architecture

```
[Pass 1: Extraction]                  [Pass 2: Validation]            [Pass 3: Synthesis]
  per-document                         claim-level                      claim-level
  ↓                                    ↓                                ↓
documents.extracted_data    →    claim_validations              →   findings + questions
  (kind=normalized_extraction)         (this sprint)                    (SPRINT-003A)
```

**Trigger:** Inngest event `claim/extraction.completed`. Per PR #59: reuse if exists, otherwise add to pass-1 finalizer.

**Failure isolation:** each of the 3 layers terminates independently as completed/skipped/failed. No layer can block another.

## 3. Per-Layer Specification

All three layers conform to TS contracts in `lib/validation/types.ts`. Each payload includes `evidence: EvidenceRef[]` for traceability.

### 3.1 Layer 11.1 — name_match

**Logic:**

1. Collect (document_id, field_path, raw_name) from name fields. Exclude `witness_letter.*`.
2. Normalize: trim, collapse whitespace, lowercase, NFC, strip diacritics.
3. Cluster. Determine canonical (priority id_document > claim_form > others).
4. Classify: `exact_match` | `fuzzy_match` (similarity ≥ 0.85) | `mismatch`.

**Reuse from PR #59:** transliteration + fuzzy match utilities (if exist). If neither — minimal helpers in `/lib/validation/string-similarity.ts`. No new npm deps.

**Output:**

```typescript
{
  canonical_name: string;
  canonical_source: { document_id: string; field_path: string };
  matches: Array<{
    document_id: string;
    field_path: string;
    raw: string;
    normalized: string;
    status: 'exact_match' | 'fuzzy_match' | 'mismatch';
    similarity: number;
  }>;
  evidence: EvidenceRef[];
  summary: { total: number; exact: number; fuzzy: number; mismatch: number };
}
```

**Skip:** zero name fields → status=`skipped`, reason=`no_name_fields`.

### 3.2 Layer 11.2 — date validation

**Reference dates:** from `claims` per PR #59.

**Rules:**

- `policy_coverage`: incident_date ∈ [policy_start_date, policy_end_date]. Outside = flag (NOT fail).
- `submission_timing`: claim_submitted_date ≥ incident_date.
- `travel_containment`: flight/hotel dates contain or overlap incident_date.
- `document_age`: report dates ≥ incident_date.

**Output:**

```typescript
{
  timeline: Array<{ document_id: string; field_path: string; date: string; role: string }>;
  rules: Array<{
    id: 'policy_coverage' | 'submission_timing' | 'travel_containment' | 'document_age';
    status: 'pass' | 'fail' | 'skipped';
    details: string;
  }>;
  evidence: EvidenceRef[];
  summary: { pass: number; fail: number; skipped: number };
}
```

**Date format:** date-only, no time component (MVP).

### 3.3 Layer 11.3 — currency validation (CORRECTED — provider deployment)

**Settlement currency:** field name on `claims` per PR #59. Default if NULL: ILS.

**Logic:**

1. Collect (document_id, field_path, amount, currency, relevant_date).
2. Convert via `ExchangeRateProvider.getRate(date, from, to)`.
3. Sum to total in settlement currency.
4. Outliers: z-score > 3 OR amount > 10× median.

**ExchangeRateProvider:**

```typescript
interface ExchangeRateProvider {
  getRate(date: string, from: string, to: string): Promise<number | null>;
}
```

**Provider implementations:**

- `FakeExchangeRateProvider`: deterministic, fixture-based. Default for unit tests, integration tests, and non-prod smoke.
- `FetchExchangeRateProvider`: live impl using `fetch` against frankfurter.app. **Implemented in this sprint but NOT enabled by default.** Activation requires:
  - Separate CEO-approved release/env decision.
  - Configuration via env var (e.g. `VALIDATION_FX_PROVIDER=live`).
  - Production hardening checklist (rate-limit handling, error budgets) — out of scope for 002C.
- Default behavior in 002C: all environments use Fake. No live FX call ships to any environment via this sprint.

**Cache:** in-memory map per function execution. Not persisted across runs.

**Failure:** provider returns null → item marked `rate_failure`. Layer continues. Does NOT block.

**Output:**

```typescript
{
  settlement_currency: string;
  items: Array<{
    document_id: string;
    field_path: string;
    raw_amount: number;
    raw_currency: string;
    rate: number | null;
    rate_date: string;
    rate_source: 'fake' | 'frankfurter' | 'cache';
    normalized_amount: number | null;
    status: 'ok' | 'rate_failure';
  }>;
  total_normalized: number;
  outliers: Array<{ document_id: string; z_score: number; reason: string }>;
  evidence: EvidenceRef[];
  summary: { total_items: number; currencies_seen: string[]; rate_failures: number };
}
```

**Codex selects ONE of two forms based on PR #59 schema findings.** Both forms enforce NOT NULL on the pass key — nullable would break UNIQUE.

**Form A — if `passes.id` exists per PR #59:**

CREATE INDEX idx_claim_validations_pass ON claim_validations(pass_id);

**Form B — if `passes.id` does NOT exist (pass_number-only) per PR #59:**

## 5. Pass Lifecycle (non-blocking layer failures)

- New `passes` row inserted at validation start: status=`in_progress`, pass_number=2 (or next per current scheme).
- Each layer writes one row to `claim_validations` with status `completed` | `failed` | `skipped`.
- **Pass row transitions:**
  - `pass.status = completed` when all 3 layers reach any terminal state (completed/skipped/failed). Includes the case where all 3 failed.
  - `pass.status = failed` ONLY when the validation handler itself cannot persist/finalize (e.g., DB connection lost, claim_validations migration not applied). NOT due to individual layer failure.
- Layer failure = data, not pass-level failure. Synthesis decides downstream.

### `lib/validation/types.ts`

```typescript
export interface EvidenceRef {
  document_id: string;
  field_path: string;
  raw_value?: string; // optional — only if value originates from documents.extracted_data
  normalized_value?: string; // optional — post-normalization form (if layer normalized it)
}

export type LayerStatus = 'completed' | 'failed' | 'skipped';

export interface NameMatchPayload {
  /* per 3.1 */
}
export interface DateValidationPayload {
  /* per 3.2 */
}
export interface CurrencyValidationPayload {
  /* per 3.3 */
}

export interface ValidationLayerResult<P> {
  layer_id: '11.1' | '11.2' | '11.3';
  status: LayerStatus;
  payload: P;
  reason?: string;
}
```

**EvidenceRef content rule:**

- `raw_value` is permitted ONLY if the value already exists in `documents.extracted_data` (i.e., already validated by extraction layer).
- NEVER store raw OCR text, raw model output, raw file content, or any unvalidated string.
- `normalized_value` stores the post-normalization form if the layer transformed the value (e.g., name normalized, currency converted).
- If neither is meaningful for an evidence row, both can be omitted; `document_id` + `field_path` is sufficient pointer.

SPRINT-003A imports from `lib/validation/types.ts`.

## 7. Inngest Integration

### Event `claim/extraction.completed`

- Per PR #59: confirm exists or create.
- If create: fire from pass-1 finalizer (path per PR #59).
- Payload: `{ claim_id }`.

### Handler `run-validation-pass`

- Subscribes to `claim/extraction.completed`.
- Steps (sequential):
  1. `create-validation-pass-row`: insert passes row.
  2. `layer-11.1-name-match`: run + write claim_validations row.
  3. `layer-11.2-date-validation`.
  4. `layer-11.3-currency-validation`.
  5. `finalize-validation-pass`: transition pass row to completed (per section 5), fire next event.

### Event `claim/validation.completed`

- Fired by step 5.
- Payload: `{ claim_id, pass_id_or_number }`.
- SPRINT-003A consumes.
